Why Rusty

A different class of agent

Most coding agents ship as Node.js or Python packages with heavy runtimes, deep dependency trees, and loose security boundaries. Rusty compiles to a single static binary with none of that baggage.

Why WindowServer CPU Matters on macOS

Web-tech developer tools use Electron or Chrome rendering layers, forcing macOS WindowServer to continuously recompute compositing arrays. Two Node/TS agent sessions can consume 50% of system CPU just to draw the interface. Rusty writes directly to the terminal via ioctl, reducing WindowServer overhead by 99%.

Philosophy

Privacy by default. Security by design.

Most coding tools require bloated runtimes, background sync, and telemetry streams. Rusty operates under an absolute privacy model: zero analytics, zero phone-home, zero backdoors.

Zero Telemetry

No analytics instrumentation, tracing pipelines, or outbound metrics. Chat payloads transit strictly between your host and the LLM endpoint you configure.

Open Source, Fully Auditable

Licensed under AGPL-3.0-or-later. Every crate is distributed as open, verifiable source code. No compiled node packages, binary blobs, or obfuscated scripts.

Path Sandboxing

Every file read, write, and edit is validated against the working directory. Parent directories (~/.ssh, /etc) are blocked. TOCTOU-hardened against symlink races.

Native Keyring Storage

API keys stored in hardware-backed OS keychains (macOS Keychain, Linux Secret Service, Windows Credential Manager) instead of plaintext .env files.

Privacy Guarantee

Zero analytics, middleware, or backdoors.

Rusty does not contact any telemetry servers. The codebase contains no trackers or analytics libraries. Requests stream directly between your terminal and the endpoint you specify.

Capabilities

Built for complex, multi-step work

Rusty coordinates multiple tools, manages task plans, and adapts its reasoning budget dynamically to context pressure.

Auto-Plan with Task Tracking

Launch with --plan-with-tasks to create a structured task list before executing. Tasks move through pending, in-progress, and completed states. Plan mode restricts to read-only for safe exploration.

Concurrent Tool Execution

Multiple independent tool calls execute in parallel via tokio::JoinSet. Results collected in call-order. Reduces wall-clock time for multi-file edits and batch searches.

Multimodal Image Support

Paste images into the TUI. Screenshots and diagrams are sanitised (ANSI, control characters stripped), encoded as base64, and sent as ContentBlock::Image to vision-capable models. 100KB paste cap.

Model Registry & Picker

Configure multiple providers (OpenAI, Anthropic, Ollama, DeepSeek, Kimi, Xiaomi) in settings.json. Switch models mid-session via /settings in the TUI.

Dynamic Thinking Budget

Three levels (minimal/normal/deep) control reasoning token allocation. The agent auto-adjusts: multi-step tasks get boosted, high context pressure triggers step-down. Set via --thinking-budget or config.

Tiered Permission System

Four modes: Default (prompt for writes), AcceptEdits (auto-allow writes), Bypass (allow all), Plan (read-only). Bash commands are auto-classified: git status runs without prompting; rm -rf requires approval.

Auto-Compaction

When messages exceed ~80k tokens or 40 messages, older messages are summarised automatically (keeping the last 10). Keeps conversations within context limits without losing critical information.

Sub-Agent Spawning

The agent tool spawns independent sub-agents in separate tokio tasks with bypass permissions. Sub-agents get all tools except agent (preventing recursive spawning).

Security

Hardened at every layer

SSRF Protection

web_fetch blocks localhost, private IPs, link-local, CGNAT, and multicast ranges. DNS rebinding prevented via IP pinning. All redirect targets re-validated.

TOCTOU-Hardened Path Sandboxing

resolve_path() avoids checking path.exists() before canonicalisation. Pre-write and post-write symlink verification guards against race conditions.

Prompt Injection Defences

Memory content sanitised on input and output. Context files escaped to prevent XML breakout. TUI paste strips ANSI, bidi overrides, and zero-width characters. 30KB context file budget prevents flooding.

File Permission Hardening

Config directories created with 0o700. Settings and memory files with 0o600. No world-readable secrets.

Installation

Two ways to install

On first run, the interactive setup wizard secures your credentials and configures provider presets. No network calls until you start a session.

1

Build from source

Clone the repository and compile a release binary. Full audit possible before running.

git clone https://github.com/PDG-Global/rusty.git
// Compile optimized release binary
cd rusty && cargo build --release
2

First-run setup wizard

Launch Rusty and the interactive wizard configures your provider, API key storage, and model selection automatically.

./target/release/rusty --setup
// Or just run `rusty` directly (wizard auto-launches)
./target/release/rusty

Sessions

Persistent conversation history

Every conversation is saved to ~/.rusty/sessions/ as JSON. Sessions are isolated per-project (derived from your working directory) and resume from where you left off.

Auto-Save on Exit

Sessions save automatically when you exit via /quit, /exit, or Ctrl+C.

Resume from CLI

Pass --resume SESSION_ID to pick up a conversation. Use --list-sessions to see all available IDs.

Interactive Picker (TUI)

Type /resume inside a session to browse saved conversations with an interactive list.

Auto-Compaction

When messages exceed ~80k tokens or 40 messages, older messages are summarised (keeping the last 10). Trigger manually with /compact.

Configuration

Settings, credentials, and presets

Configuration resolves through a layered chain: environment variables take highest precedence, then CLI flags, then the settings file. The setup wizard populates ~/.rusty/settings.json on first run.

The allowed_tools array is populated by permanent permission grants during sessions.

API Key Resolution Chain

Credentials are resolved in this priority order. The first non-empty value found is used:

1
RUSTY_API_KEY
Environment variable. Highest priority.
2
OPENAI_API_KEY
Environment variable. Standard OpenAI-compatible key.
3
OS Keyring
macOS Keychain, Linux Secret Service, or Windows Credential Manager.
4
Settings File
The api_key field in ~/.rusty/settings.json.
